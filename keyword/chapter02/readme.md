### SQL 쿼리(CRUD, 필터링, 정렬, 제한, 집계 등 기본적인 SQL 문법을 정리해주세요)
    
    **SQL** (**S**tructured **Q**uery **L**anguage)이란?
    
    → 관계형 데이터베이스관리 시스템(RDBMS)에서 정보를 저장하고 처리하기 위해 설계된 언어

    - **SQL의 문법은 크게 4가지로 분류된다 : DDL, DML, DCL, TCL**
    1. **DDL** (**D**ata **D**efinition **L**anguage)
    : 테이블과 컬럼을 정의하는 명령어 
        
        → 생성, 수정, 삭제 등의 데이터 전체 골격을 결정하는 역할
        
        - **특징**
        명령어를 입력하는 순간 작업이 즉시 반영되기 때문에 사용 시 주의해야 한다
        - **종류**
        CREATE : 테이블 생성
        ALTER : 테이블 구조 수정
        DROP : 테이블 삭제
        RENAME : 테이블 이름 변경
        TRUNCATE : 테이블 초기화
        + ADD(컬럼 추가), MODIFY(컬럼 수정)
    2. **DML** (**D**ata **M**anipulation **L**anguage)
    : DB의 내부 데이터를 관리하기 위한 명령어 
        
        →  조회, 추가, 변경, 삭제 등의 작업을 수행
        
        - **특징**
        DDL과 달리 즉시 반영이 되지 않는다. 따라서 ROLLBACK으로 되돌릴 수 있다
        타겟 테이블을 메모리 버퍼 위에 올려두고 수행하기 때문에, 커밋을 통해 트랜잭션을 종료해야 변경사항이 반영된다
        - **종류**
        SELECT : 데이터 검색
        INSERT : 데이터 추가
        UPDATE : 데이터 수정
        DELETE : 데이터 삭제
        
    3. **DCL** (**D**ata **C**ontrol **L**anguage)
    : 데이터를 관리하며 보안, 무결성, 회복, 병행 제어 등을 정의하는 데 사용하는 명령어
        
        → 읽기, 쓰기 권한을 부여/박탈할 수 있고, 트랜잭션을 명시/조작할 수 있음
        
        - **특징**
        데이터를 보호하기 위한 보안 역할 수행한다
        데이터 정확성을 위해 무결성을 유지하며, 시스템 장애에 대비한 회복과 병행수행을 제어한다
        - 종류
        GRANT : 권한 정의
        REVOKE : 권한 삭제
        
    4. **TCL** (**T**ransaction **C**ontrol **L**anguage)
    : DCL과 비슷하지만, 데이터 제어가 아닌 트랜잭션을 제어할때 사용
        
        → 작업 단위를 묶어 DML에 의해 조작된 결과를 트랜잭션 별로 제어한다
        
        - **종류**
        COMMIT : 작업을 정상적으로 처리하겠다는 명령어
        ROLLBACK : 작업을 되돌려 놓겠다는 명령어
        SAVEPOINT : 커밋 전 특정 시점까지만 반영하거나 롤백하겠다는 명령어

    
    - **CRUD (C**reate**, R**ead**, U**pdate**, D**elete**)**
        
        > DB 관리 및 응용 프로그래밍 개발에서 기본적으로 사용되는
        > 
        > 
        > 네 가지 기본 프로세스
        > 
        - **Create** : 데이터 추가 (INSERT문 사용)
        
        ```sql
        INSERT INTO member (name, age)
        VALUES (세라, 100);
        ```
        
        - **Read** : 데이터 조회 (SELECT문 사용)
        
        ```sql
        SELECT * FROM member;    #전체 조회
        
        SELECT name, age FROM member    #특정 컬럼(이름, 나이)만 조회
        
        SELECT * FROM member    #50세 이상만 조회
        WHERE age >= 50;
        ```
        
        - **Update** : 데이터 수정 (UPDATE문 사용)
        
        ```sql
        UPDATE member
        SET age = 101
        WHERE id = 1;    #WHERE절이 없으면 모든 데이터가 수정됨
        ```
        
        - **Delete** : 데이터 삭제 (DELETE문 사용)
        
        ```sql
        DELETE FROM member
        WHERE id = 1;
        ```
        
        ⭐**중요성**⭐
        
        1. 데이터를 효율적으로 관리•유지할 수 있다
        2. 대부분의 데이터 중심 프로그램은 CRUD 기능 기반으로 구축된다
        3. 데이터를 동적으로 생성•조회•수정•삭제할 수 있다
        4. 데이터 조작에 대한 표준화된 접근 방식을 제공한다 (개발자들이 공통된 용어와 방식을 사용하게 되기 때문)
        
        → 핵심 구성요소이며, 정보 처리과정에서의 기본 도구이다.
        
    - **필터링 (WHERE)**
        
        > 특정 조건에 맞는, 필요한 데이터만 골라내는 기능
        > 
        > 
        > 여러 방법이 있지만, WHERE절을 많이 이용한다
        > 
        
        **주요 조건식** **(WHERE)**
        
        | **조건식** | **의미** |
        | --- | --- |
        | = | 같다 |
        | != , <> | 다르다 |
        | >, >=, <, <= | 비교 |
        | BETWEEN A AND B | 범위 |
        | IN (...) | 여러 값 중 하나 |
        | LIKE ‘%문자%’ | 문자열 검색 |
        | IS NULL | NULL 여부 |
        
        ```sql
        #다양한 조건식
        WHERE age >= 50
        WHERE age BETWEEN 10 AND 50
        WHERE id IN (1,2,3)
        WHERE name LIKE '%세라%'
        WHERE age IS NULL
        ```
        
        → 원하는 데이터만 정확하게 추출할 수 있다!
        
    - **정렬 (ORDER BY)**
        
        > 조회 결과를 특정 기준으로 정렬하는 기능
        > 
        > 
        > SQL은 기본적으로 SELECT로 검색된 결과를 테이블에 입력된 순서대로 출력하지만, 특정 컬럼을 시준으로 오름차순, 내림차순을 적용해야 할 때 사용한다
        > 
        
        **ASC** : 오름차순 , **DESC** : 내림차순
        
        ```sql
        SELECT *
        FROM member
        ORDER BY age DESC;    #나이를 내림차순으로 정렬
        ```
        
    
    - **제한 (LIMIT)**
        
        > 조회된 데이터 출력 시
        > 
        > 
        > 몇 개의 행을 반환할 것인지, 개수를 제한할 때 사용하는 기능
        > 
        > SELECT문의 마지막에 지정한다
        > 
        
        ```sql
        SELECT *
        FROM member
        LIMIT 10;
        
        SELECT *
        FROM member OFFSET 20;    #21번부터 출력됨
        ```
        
    - **집계 (Aggreate)**
        
        > 여러 행으로부터 하나의 결과값을 반환하는 함수
        > 
        > 
        > SELECT문에서만 사용되며, 합, 최대/최소, 평균 등을 구하는 데 사용된다
        > 
        - **COUNT( )**
            
            : 특정 컬럼의 행의 개수를 세는 함수
            
            COUNT(*) → 테이블에 존재하는 행의 개수 반환
            
            COUNT(특정 열) → 해당 열이 NULL이 아닌 행의 개수 반환
            
            DISTINCT를 이용하면 중복을 제외한 값의 개수 구할 수 있음
            
        - **AVG( ) / SUM( )**
            
            : 행 안에 있는 값의 평균/합을 계산하는 함수
            
            NULL값은 무시하고 계산 → NULL ≠ 0
            
        - **MAX( ) / MIN( )**
            
            : 행 안에 있는 값의 최댓값/최솟값을 구하는 함수
            
    - **그룹화 (GROUP BY)**
        
        > 특정 열을 기준으로 데이터를 그룹화함
        > 
        > 
        > 주로 집계 함수와 같이 사용된다
        > 
        
        - **HAVING**
            
            : GROUP BY와 함께 사용되며, 그룹하된 결과에 조건을 적용한다
            
            (WHERE절과 비슷한 역할을 한다)
            
    - **테이블 연결 (JOIN)**
        - INNER JOIN
            
            공통된 데이터만 조회한다
            
        - LEFT JOIN
            
            왼쪽 테이블을 기준으로 모두 조회한다
            
    
    **SQL은 작성 순서와 실행 순서가 다르다**
    
    **실행 순서**
    
    **: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT**


### 트랜잭션이란 무엇인가?
    
    DB를 다루다 보면 단순히 데이터를 조회하거나 수정하는 것을 넘어서, 여러가지 작업을 하나의 묶음으로 처리해야 하는 상황이 발생한다
    
    → 이 때 사용하는 개념이 Transaction이다
    
    **Transaction이란?**
    
    > DB에서 하나의 논리적인 작업 단위
    > 
    > 
    > 여러 개의 SQL문이 하나의 작업처럼 묶여서, 모두 성공하거나 모두 실패해야 하는 실행 단위이다
    > 
    
    **Transaction이 필요한 이유**
    
    : 데이터의 정확성과 일관성이 중요하기 때문
    
    예를 들어, A계좌에서 B계좌로 10만원을 송금할 때
    
    1. A계좌에서 10만원 감소
    2. B계좌에서 10만원 증가
    
    ㄴ 위의 두 작업이 모두 수행되어야 정상적인 이체가 된다
    
    만약, A계좌에서 돈이 빠졌는데, 중간에 문제가 생겨서 B계좌에 돈이 안 들어갔다면 큰일이다
    
    그러므로, 이 두 작업을 하나로 묶어서 하나라도 실패한다면 롤백을 해서 데이터 일관성을 유지할 수 있다! 
    
    → 계좌 이체 상황 말고도, 쇼핑몰 결제 처리, 주문 생성+재고 감소 등 여러 곳에서 쓰인다
    
    **특성**
    
    1. 원자성 (Atomicity)
        
        트랜잭션의 연산은 DB에 전부 반영이 되거나, 전부 반영이 되지 않아야 한다
        
    2. 일관성 (Consistency)
        
        시스템이 가지고 있는 고정요소는 트랜잭션 수행 전후의 차이가 없어야 한다
        
    3. 독립성 (Isolation)
        
        수행중인 트랜잭션이 완전히 끝날 때까지 다른 트랜잭션에서 수행 결과를 참조할 수 없다
        
    4. 영속성 (Durability)
        
        성공적으로 완료된 트랜잭션의 결과는, 시스템이 고장나더라도 영구적으로 반영되어야 한다
        
    
    ⭐**중요성**⭐
    
    1. 데이터 무결성 보장 : 데이터가 깨지거나 잘못 저장되는 것을 방지한다
    2. 안정적인 시스템 운영 : 예외 상황에서도 데이터가 안전하게 유지된다
    3. 동시성 제어 : 여러 사용자가 동시에 작업할 때, 충돌을 방지한다
    
    → 트랜잭션은 DB의 신뢰성을 보장하는 핵심 개념이다

### 인덱스(Index)란?
    
    > 데이터를 빠르게 찾기 위해 사용되는 자료구조
    > 
    > 
    > 테이블의 특정 컬럼에 대해 검색 속도를 향상시키기 위한 별도의 구조를 생성하는 것 (목차 역할)
    > 
    
    인덱스가 없다면 모든 데이터를 하나씩 확인해야 해서, 데이터가 많은 경우에는 느려진다
    
    **💥장점**
    
    1. 정렬된 상태를 유지한다
    2. 빠른 탐색이 가능하다
    3. 범위 검색에 유리하다
    4. =, <, > 같은 조건에서 특히 빠르다
    
    **❎단점**
    
    1. INSERT, UPDATE, DELETE가 느려진다 : 인덱스도 같이 수정되어야 하기 때문
    2. 저장 공간 추가 사용 : 인덱스 자체도 데이터이기 때문
    3. 너무 많으면 성능 저하 : 관리 비용이 추가되기 때문
    
    → 인덱스는 검색 속도를 빠르게 하기 위한 데이터의 목차이다
    
    → 무조건 많이 쓰는게 아니라, 필요한 곳에 잘 써야 한다


### ORM VS Raw SQL 의 차이를 정리하고 유명한 ORM을 정리해주세요
    
    **ORM : 객체를 통해 DB를 다루는 방식**
    
    > 객체와 DB 테이블을 매핑해서 SQL문을 직접 작성하지 않고 DB를 다루는 방식이다
    > 
    > 
    > SQL 대신 코드로 데이터를 조작하기 때문에
    > 
    > → 테이블을 클래스처럼 사용하고, 테이블을 객체처럼 다룰 수 있게 해 준다
    > 
    
    **장점**
    
    코드 개발 속도가 향상된다
    
    코드 가독성이 증가한다
    
    이식성이 좋아서 DB 독립성이 증가한다
    
    객체지향적인 개발이 가능하다
    
    **단점**
    
    성능 최적화가 어렵다
    
    복잡한 쿼리 작성하기가 불편하다
    
    내부 SQL 동작을 모른다면 비효율적이다
    
    **Raw SQL : SQL을 직접 작성하는 방식**
    
    > SQL문을 직접 작성해서 DB를 제어하는 방식이다
    > 
    > 
    > 개발자가 SELECT, INSERT, UPDATE, DELETE 등을 직접 작성해서 실행하며,
    > 
    > DB와 가장 직접적으로 상호작용하는 방식이다
    > 
    
    **장점**
    
    직접 쿼리를 제어할 수 있으므로 성능 최적화에 유리하다
    
    JOIN, 서브쿼리 등을 자유롭게 사용하며 복잡한 쿼리를 작성할 수 있다
    
    개발자가 DB 동작을 정확하게 이해할 수 있다
    
    **단점**
    
    코드가 길어지고 가독성이 떨어진다
    
    유지보수가 어렵다
    
    DB가 바뀌면 쿼리를 수정해야 하기 때문에 DB 종속성이 증가한다
    
    **ORM VS Raw SQL**
    
    |  | ORM | Raw SQL |
    | --- | --- | --- |
    | 사용 방식 | 객체 기반 | 직접 작성 |
    | 난이도 | 쉬움 | 어려움 (상대적) |
    | 생산성 | 높음 | 낮음 |
    | 성능 | 제한적 | 최적화 가능 |
    | 유연성 | 제한적 | 매우 높음 |
    | 유지보수 | 쉬움 | 어려움 |
    
    **ORM의 종류**
    
    - Java
        - Hibernate
        - Spring Data JPA
    - Pyrhon
        - Django ORM
        - SQLAlchemy
    - JavaScript (Node.js)
        - Sequelize
        - TypeORM
        - Prisma
    - Ruby
        - ActiveRecord
