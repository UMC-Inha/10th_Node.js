# 시니어 미션

# API 디자인 모범사례

https://learn.microsoft.com/ko-kr/azure/architecture/best-practices/api-design - 문서를 읽고 주요 내용을 간단히 정리해주세요.

- 정리
    - **RESTful API**
        
        > **REST**(**Re**presentaional **S**tate **T**ransfer) 아키텍처 원칙을 따르는 웹 API
        > 
        
        HTTP 프로토콜을 기반으로 >리소스 중심 설계 + 표준 메서드< 사용
        
        - **특징**
            - 플랫폼 독립성 : 내부 구현에 관계없이 클라이언트가 API를 호출할 수 있음
                
                                         - HTTP를 표준 프로토콜로 사용
                
                                         -  JSON/XML 형식으로 데이터 전달
                
            - 느슨한 결합 : 클라이언트와 웹 서비스가 따로 발전할 수 있음
                
                                     - 클라이언트 ↔ 서버 - 서로의 내부 구현을 알 필요 X
                
                                     - 표준 프로토콜만 사용
                
    - **핵심 개념**
        - **URI**(**U**niform **R**esource **I**dentifier)
            
            : 리소스를 고유하게 식별할 수 있는 식별자, 모든 리소스는 URI로 표시됨
            
        - **리소스 표현**
            
            : URL로 식별되는 리소스를 XML/JSON 같은 형식으로 데이터를 전송
            
            {URI} GET 요청 수행 → JSON 본문 받음
            
        - **균일한 인터페이스**
            
            : RESTful API가 클라이언트와 서비스 구현 간의 느슨한 결합을 달성하는 방법
            
            리소스에 대해 GET, POST, … 같은 작업을 수행하기 위해 표준 HTTP 동사 사용
            
        - **상태 비지정 요청 모델**
            
            : HTTP 요청은 독립적이며 어떤 순서로든 발생할 수 있음
            
            각 요청은 원자성 작업이어야 함(리소스 자체에만 정보가 저장되기 때문)
            
        - **하이퍼미디어 링크**
            
            : 응답에 대한 요청을 보낼 때 json 외에 사용하는 새로운 타입의 데이터
            
            REST API는 이 링크에 의해 구동될 수 있음
            
    - **리소스 URI 이름 규칙**
        1. 동사 대신 명사 사용
        2. 복수 명사로 컬렉션 이름 지정
        3. 리소스 간의 관계와 이를 노출할 방법 고려
        4. 단순하고 유연한 관계 유지
        5. 적은 수의 리소스를 많이 사용하지 않기(큰 리소스로 결합하는게 좋지만 너무 하면 필요없는 데이터를 가져오게 되므로 둘 사이의 균형 유지)
        6. DB 내부 구조 미러링 금지
        
    - **HTTP 요청 메서드**
        
        GET, POST, PUT, PATCH, DELETE
        
        | GET | 리소스 검색 |
        | --- | --- |
        | POST | 리소스 생성, 기존 리소스에 처리하기 위한 데이터 제출 |
        | PUT | 기존 리소스 전체 업데이트, 리소스 생성(존재하지 않는 경우) |
        | PATCH | 기존 리소스 부분 업데이트 |
        | DELETE | 리소스 제거 |
    - **리소스 MIME 형식**
        
        > **MIME**(**M**ultipurpose **I**nternet **M**ail **E**xtentions), 미디어 형식으로도 알려져 있음
        > 
        - 슬래시 (/) 로 구분된 type/subtype 구성
            
            type : video, text와 같은 일반 카테고리
            
            subtype : 정확한 데이터 종류를 식별
            
                      text 하위 타입 →  plain, html, calendar
            
              이처럼 각 타입에는 고유한 하위  타입이 있음
            
        
    - **비동기 메서드**
        
        경우에 따라 메서드를 완료하는 데 시간이 걸리는 처리가 필요해서 대기 시간이 발생할 때, 메서드를 비동기식으로 만드는 것이 좋음
        
        → HTTP 상태 코드 202(수락됨)를 반환하여 요청 처리가 허용되었지만 불완전함을 나타내기
        
    - **부분 응답**
        
        리소스에 파일이나 이미지와 같은 큰 용량의 이진 필드가 포함될때, 불안정하고 일시적인 연결로 인한 문제를 극복하고 응답 시간을 개선하기 위해 지원하는 것이 좋다
        
        GET 요청에 대한 Accept-Ranges 헤더를 지원해서, 해당 작업이 부분 요청을 지원한다는 것을 나타내기 → 클라이언트는 리소스의 하위 집합을 반환하는 GET 요청을 보낼 수 있음
        
        클라리언트가 HEAD 요청을 실행해서 부분 GET 요청을 통해 리소스를 가져올지 여부를 결정할 수 있음
        
    - **HATEOAS (H**ypertext **a**s **t**he **E**ngine **o**f **A**pplication **S**tate**)**
        
        → API 응답 안에, 현재 상태에서 가능한 다음 행동 링크들을 포함하는 것
        
        각 HTTP GET 요청은 
        
        1. 응답에 포함된 하이퍼링크를 통해 요청된 개체와 직접 관련된 리소스를 찾는 데 필요한 정보를 반환해야 한다.
        2. 각 리소스에서 사용할 수 있는 작업에 대한 정보를 제공해야 한다.
        
        장점
        
        - 클라이언트가 유연하게 처리 가능(가능한 행동을 응답에 포함시키기 때문)
        - 클라이언트가 로직을 알 필요가 없음(서버가 안내하는 링크만 따라가면 되기 때문)
        - 유지보수성 향상(클라이언트는 링크만 따라가니까)
        - 자동 탐색 가능(문서가 없어도 클라리언트, 자동화 도구가 API 탐색을 할 수 있음)
        
    - **버전 관리**
        
        웹 API는 정적 상태로 유지되지 않기 때문에 버전 관리 사용
        
        - 버전 관리 X
            
            : 변경 내용은 새로운 리소스/링크로 나타낸다
            
            - 필드 제거나 이름을 바꿀 때 리소스 스키마를 크게 수정해야 한다
            - 호환성이 손상될 수 있다
        - URI 버전 관리
            
            : 수정할 때마다 각 리소스의 URI에 번호를 추가한다
            
            - API가 여러번 반복을 통해 완성되고, 서버가 다양한 버전을 지원 →  다루기 어려울 수 있음
            - 모든 링크에 버전 번호를 포함해야 함 → HATEOAS 구현을 복잡하게 만든다
        - 쿼리 문자열 버전 관리
            
            : HTTP 요청에 추가된 쿼리 문자열 내 매개 변수를 사용
            
            - 리소스가 항상 동일한 URL에서 검색됨
            - 쿼리 문자열을 분석하고 적절한 HTTP응답을 다시 보냄 → HATEOAS 구현을 복잡하게 만든다
        - 헤더 버전 관리
            
            : 리소스의 버전을 나타내는 지정 헤더를 구현함
            
            - 클라이언트 앱이 모든 요청에 적절한 헤더를 추가해야 함
            - HATEOAS를 구현하려면 모든 링크에 적절한 사용자 지정 헤더를 포함해야 함 → 복잡
        - 미디어 유형 버전 관리
            
            : 서버에 GET 요청을 보낼 때 Accept 헤더를 사용
            
    - **다중 테넌트 웹 API**
        
        다중 테넌트는 단일 웹 API 내의 여러 테넌트에서 리소스에 액세스하고 검색하는 방법을 결정 → API 디자인에 큰 영향을 준다
        
        시스템 내의 모든 사용자에게 일관적이면서도 유연한 환경을 보장함
        
        - 하위 도메인 또는 도메인 기반 격리 (DNS 수준 테넌트)’
            
            테넌트별 도메일은 사용해 요청을 라우팅함
            
            내부 URL이 노출되지 않기 위해 호스트 이름을 유지함
            
            테넌트별 트래픽의 올바른 라우팅 보장, 내부 인프라 보호
            
        - 테넌트별 HTTP 헤더 전달
            
             캐시 계층이 URI 기반 키만 사용하고 헤더를 고려하지 않는 경우 캐싱이 복잡하다
            
             응답을 구분하지 않는 경우 한 테넌트를 위한 캐시된 데이터를 다른 테넌트에 제공하고 데이터 유출 위험을 초래할 수 있다
            
        - URI 경로를 통한 테넌트별 정보 전달
            
            리소스 계층 내에 테넌트 식별자를 추가하고 API 게이트웨이 또는 역방향 프록시를 사용하여 경로 세그먼트에 따라 적절한 테넌트를 결정
            
            RESTful 디자인을 손상시키고 더 복잡한 라우팅 논리를 도입함
            
    - **웹 API 완성도 모델**
        - 수준 0 : 하나의 URI 정의, 모든 작업은 해당 URI에 대한 POST 요청
        - 수준 1 : 개별 리소스에 대한 별도의 URI 생성, 아직 RESTful 아님
        - 수준 2 : HTTP 메서드를 사용해서 리소스에 대한 작업 정의, 많은 웹 API가 이 수준
        - 수준 3 : HATEOAS를 사용, RESTful 수준

# API 버저닝

실제로 서비스를 배포하다보면 서버를 항상 띄워놓고 새로운 버전의 API를 도입해야하는 경우가 있습니다. 이런 상황에서 어떤 전략을 사용할 수 있는지 고민해서 정리해주세요

- 정리
    
    다양한 클라이언트가 해당 API를 사용하고 있을 때 새로운 버전을 도입하기 위해서는, 기존 사용자에게 영향을 주지 않게 하는 것이 중요함
    
    기존 API를 유지하지 않은 채로 변경을 하게 되면 오류가 발생하기 때문에, 기존 API를 유지하면서 새로운 버전을 병행해서 제공해야 한다 → 기존 버전을 제어하는것보다, 일정 기간 동안 두 버전을 동시에 운영하기 → 새 버전에 대한 알림을 사용자에게 알리고, 새로운 버전으로 옮겨갔을 때 이전 버전 삭제
    

# 멱등적 설계

네트워크 오류나 중복 요청에도 데이터가 꼬이지 않도록 어떻게 API를 설계할 수 있을까요?

 https://docs.tosspayments.com/blog/what-is-idempotency → 다음 문서를 보고 정리해주세요 

- 정리
    
    **핵심 : 멱등성을 보장한다**
    
    같은 요청이 여러번 들어와도, 한 번 실행한 것과 동일해야 함
    
    1. **POST/PATCH와 같은 API는 별도로 처리**
        
        GET, PUT, DELETE와 달리 POST, PATCH는 요청할 때마다 결과가 달라질 수 있음
        
         → 중복 데이터 발생!
        
        : 이런 API에는 멱등성 보장 로직을 추가한다
        
    2. **멱등키 사용**
        
        UUID 등과 같은 고유한 키를 요청마다 헤더에 담아서 보내면, 서버는 해당 키를 기준으로 이미 처리된 요청인지를 판단할 수 있다
        
    3. **멱등키 유효기간 지정**
        
        멱등키를 계속 가지고 있으면 부담이 커짐 → 일정 시간 이후로 삭제하기
        
        이러면 나중에는 같은 키라도 새로운 요청으로 처리할 수 있다
        
    4. **요청 동일성 판단**
        
        요청 동일성을 판단할 때에, 단순히 키만 보는게 아니라 API 경로나 HTTP 메서드 등을 함께 비교해서 완전히 같은 요청인지 판단한다
        
    5. **예외 상황도 명확하게 처리**
        - 400 Bad Reauest : 멱등키 x, 형식 오류
        - 409 Conflict : 이전 요청을 처리중
        - 422 Unprocessible Entity : 같은 키인데 요청 내용이 다름
        
        위와 같이 설계하면 클라이언트가 어떻게 재시도해야할지 명확하게 알 수 있다
        
    
    **→ 멱등키 기반 설계 적용 시, 해당 상황에서도 안전하게 재요청 가능하고 데이터 중복/불일치 방지 가능**
