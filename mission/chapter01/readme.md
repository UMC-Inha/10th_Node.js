## 시니어 미션

### 일반(주니어) 미션에서 제외되었던 부분까지 ERD를 설계해 주세요. 지도 검색, 내 포인트 관리, 알림, 사장님의 자신의 점포 관리하는 부분이 포함됩니다.

### 미션 자료로 제공된 피그마를 보고 ERD를 설계한 후 제 1,2,3 정규화를 통해 제 1,2,3 정규형을 만들고 각각 중복된 데이터가 어떻게 변화하였고 어떠한 이점이 있었는 지 작성하여 주세요

1. 제 1 정규화 (1NF)

 회원가입 시에, 사용자가 선호 음식을 여러 개 선택 가능하다. 이를 하나의 컬럼에 ‘한식, 중식, 일식, …’과 같이 저장할 경우 다중값 속성이 되어서 1NF를 만족하지 못한다. <br />
 이를 해결하기 위해서 User_Food 테이블을 별도로 분리하여 행을 나눠서 저장하도록 해서, 하나의 행에는 하나의 foodtype만 저장하도록 했다. <br />
 이것을 통해 데이터의 원자성이 보장되며, 특정 음식에 대한 검색 및 관리가 용이해진다. 또한, 확장성도 증가한다! <br />

2. 제 2 정규화 (2NF)
 User_Mission 테이블은 사용자 ↔️ 미션 간 관계를 나타내는데, 미션에 대한 정보 (store_id …)는 미션 자체에 종속되는 속성이다. 이를 User_Mission 테이블에 함께 저장할 경우에는, 기본키의 일부에만 종속되는 부분종속 문제가 발생할 수 있다. <br />
 이를 해결하기 위해서 미션 관련 정보는 Mission 테이블로 분리하고, User_Mission은 사용자 ↔️ 미션 관계 및 상태 정보만을 관리하도록 했다. <br />
 이로 인해 데이터 중복이 감소하고, 특정 미션 정보가 변경될 경우, 한 곳에서만 수정하면 되기 때문에 수정 이상을 방지할 수 있다. <br />

3. 제 3 정규화 (3NF)
 주소 정보가 address, city(시), district(구), neighborhood(동), (+위도, 경도) 로 구성되어 있는데, 이것들은 서로 의미적으로 종속 관계를 가질 수 있다. <br />
 이를 제거하기 위해 주소 정보를 별도의 Address 테이블로 분리하고, Store테이블에서 address_id를 참조하도록 설계할 수 있다. <br />
 이를 통해 데이터의 일관성을 유지하고, 불필요한 중복을 최소화할 수 있다. <br />
 (하지만 조회 성능과 사용 편의성을 고려해 일부 중복을 허용하였다!) <br/>

### 피그마의 홈 부분에서 한 사람이 “미션 도전!” 버튼을 빠르게 여러 번 눌렀을 때 여러 가지 이유(비동기 로직 등)로 요청이 지연되어 완전히 처리하기 전 두 번 요청이 들어갈 수 있습니다. 이를 해결할 수 있는 방법에 대해 작성하여 주세요 (ERD 직접적으로 관련이 있기보다는 설계할 때 한번쯤 고민해보면 좋을 것 추가시켜 놓았습니다) (다양한 방법이 있으니 찾아봐 주세요)

1. 프론트엔드
 먼저, 프론트엔드에서는 사용자가 버튼을 누르는 즉시 버튼 비활성화 처리를 통해 중복 요청을 막을 수 있다. 하지만 버튼을 비활성화해도 네트워크 지연 등으로 요청이 여러번 들어갈 수 있다. 

2. 백엔드
 백엔드에서는 해당 사용자가 동일 미션에 참여 중인지 확인하는 조회 과정을 먼저 수행한다! 데이터를 조회하여 이미 존재하는 경우, 추가적인 요청을 차단하고 “이미 참여 중인 미션입니다”와 괕은 응답을 반환할 수 있다. <br />

 하지만, 사용자가 버튼을 매우 빠르게 누른다면, 거의 동시에 두 개 이상의 요청이 도달할 수 있다. 이때의 조회 시점에서는 사용자가 해당 미션에 참여중이지 않은 것으로 판단되어서, 이 로직을 통과할 가능성이 있다.

3. DB
 앞선 두 방안으로 중복 요청을 줄일 수 있지만, DB 수준에서의 제약조건 설정이 필요하다. User_Mission 테이블의 user_id와 mission_id에 대해 UNIQUE 제약조건을 설정한다. 이 조건은 동일한 id를 가지는 데이터가 두 번 이상 들어오는 것을 물리적으로 차단한다. 

- 이와 같은 방법들을 사용하면, 중복 데이터 발생을 원천적으로 차단할 수 있으며, 시스템의 데이터 무결성을 안정적으로 보장할 수 있다!
